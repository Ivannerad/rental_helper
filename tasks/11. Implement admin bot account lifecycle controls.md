# Why
Automation must be startable and stoppable per account so operations can be controlled safely.

# Context
- Lives in `src/admin_bot/` and account repositories.
- Current behavior: no lifecycle controls exist for managed accounts.
- Expected behavior: admins can start, stop, and inspect automation status per account.

# Constraints
- State changes must persist so restarts do not lose lifecycle status.
- Disabled accounts must not continue receiving or sending automated messages.
- Status commands should surface recent errors if available.

# Acceptance Criteria
- Admin commands can start and stop automation for a chosen account.
- Account status is persisted and queryable.
- Worker processes can consume the persisted status later.

# Validation
- Run tests for lifecycle state changes and persistence.
- Manually toggle an account and verify the displayed status updates correctly.

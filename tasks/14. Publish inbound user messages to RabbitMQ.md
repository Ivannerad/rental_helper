# Why
Incoming tenant messages must be normalized and forwarded to the chatbot service for processing.

# Context
- Lives in `src/userbot_worker/` and shared queue packages.
- Current behavior: inbound Telegram messages are not captured.
- Expected behavior: direct user messages become typed inbound queue events with correlation and account metadata.

# Constraints
- Only intended user conversations should be ingested; ignore irrelevant Telegram events.
- Preserve enough message metadata for tracing and deduplication.
- Do not block the Telethon event loop on slow downstream processing.

# Acceptance Criteria
- Direct user messages are transformed into inbound event payloads.
- Events are published to RabbitMQ with account ID, user ID, message text, message IDs, timestamps, and correlation data.
- Duplicate handling strategy is represented in the event metadata.

# Validation
- Run integration tests for Telegram event ingestion to queue publication.
- Manually send a test DM and confirm the event appears on the inbound queue.

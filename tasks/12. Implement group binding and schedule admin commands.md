# Why
Each account needs one bound public group and admins need visibility into upcoming viewings.

# Context
- Lives in `src/admin_bot/`, listing modules, and appointment repositories.
- Current behavior: no group binding or schedule view exists.
- Expected behavior: admins can bind one public group per account and view pending or upcoming appointments.

# Constraints
- Enforce exactly one bound group per account in v1.
- Schedule output must identify account, listing, user, and time window clearly.
- Group binding changes should not orphan existing listing records silently.

# Acceptance Criteria
- Admin commands can bind or update a public group for an account.
- Admin commands can list pending and upcoming viewing appointments.
- Validation prevents invalid or ambiguous group assignments.

# Validation
- Run tests for group binding rules and schedule queries.
- Manually bind a test group and retrieve an empty or populated schedule response.

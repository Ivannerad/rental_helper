# Why
Distributed message handling across Telegram, RabbitMQ, and the LLM provider needs safeguards for duplicates and failures.

# Context
- Lives across shared logging, messaging, and service middleware modules.
- Current behavior: no structured logging, retry policy, or idempotency enforcement exists.
- Expected behavior: services emit correlated logs, avoid duplicate replies, and retry transient failures safely.

# Constraints
- Correlation IDs must flow from inbound message to outbound user action.
- Idempotency must prevent duplicate sends during reconnect or redelivery.
- Retry behavior must not create infinite loops or silent message loss.

# Acceptance Criteria
- Structured logs include account, conversation, and correlation identifiers.
- Idempotency checks exist for inbound and outbound event processing.
- Retry and dead-letter policies are applied to Telegram and LLM transient failures.

# Validation
- Run tests for duplicate event handling and retry classification.
- Manually trigger a redelivery or transient failure and confirm only one final user-visible reply is sent.

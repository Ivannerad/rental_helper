# Why
Each service needs a reusable producer/consumer layer for publishing and handling typed events reliably.

# Context
- Lives in a shared messaging package and service wiring modules.
- Current behavior: no RabbitMQ integration exists.
- Expected behavior: services can publish, consume, ack, retry, and dead-letter events consistently.

# Constraints
- Keep connection management and channel lifecycle centralized.
- Support retry and dead-letter routing without duplicating broker logic in business handlers.
- Do not bind business code directly to broker-specific details where avoidable.

# Acceptance Criteria
- A shared transport layer manages RabbitMQ connections and queue bindings.
- Consumers can register handlers for inbound and outbound event types.
- Dead-letter behavior is wired for malformed or exhausted messages.

# Validation
- Run integration tests against RabbitMQ for publish/consume/ack flows.
- Manually confirm dead-letter routing with an intentionally invalid payload.

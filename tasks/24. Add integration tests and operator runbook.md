# Why
The full system needs end-to-end verification steps for local development and future deployment.

# Context
- Lives in `tests/`, service wiring, and operational documentation.
- Current behavior: there are no integration tests or operator instructions.
- Expected behavior: contributors can verify queue flow, admin commands, and core conversation scenarios in a repeatable way.

# Constraints
- Integration tests should use local or containerized dependencies rather than production services.
- The runbook must describe setup, startup order, and manual smoke tests.
- Keep operator instructions aligned with implemented admin bot commands.

# Acceptance Criteria
- Integration tests cover RabbitMQ flow, userbot ingestion, outbound actions, account lifecycle, group binding, and schedule retrieval.
- A runbook documents local startup and manual acceptance scenarios.
- Manual validation steps reflect the Russian conversation flow from the implementation plan.

# Validation
- Run the integration test command against local dependencies.
- Follow the runbook manually from environment setup through a complete smoke test.

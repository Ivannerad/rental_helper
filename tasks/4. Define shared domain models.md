# Why
Core entities must be defined early so all services use the same language and IDs.

# Context
- Lives in shared domain modules under `src/`.
- Current behavior: managed accounts, conversations, listings, and appointments exist only in the plan.
- Expected behavior: shared models define the fields and invariants for v1 entities.

# Constraints
- Keep models framework-agnostic so they can be reused by bots, workers, and persistence layers.
- Preserve the v1 constraint of one bound public group per managed account.
- Model names should match the implementation plan terminology.

# Acceptance Criteria
- Models exist for `ManagedAccount`, `BoundGroup`, `Conversation`, `LeadRequirements`, `Listing`, `ListingOffer`, and `ViewingAppointment`.
- IDs and status fields are defined consistently.
- The model package is imported by at least one service-facing module.

# Validation
- Run unit tests for model creation and validation rules.
- Manually review fields against `IMPLEMENTION_PLAN.MD`.

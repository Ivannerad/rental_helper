# Why
Services need a clean persistence interface rather than direct SQL spread throughout the codebase.

# Context
- Lives in shared persistence modules under `src/`.
- Current behavior: no repositories exist for database access.
- Expected behavior: repositories encapsulate CRUD and query operations for accounts, conversations, listings, offers, and appointments.

# Constraints
- Repository APIs should match service use cases, not expose arbitrary database details.
- Keep transaction boundaries explicit for multi-step updates.
- Do not duplicate query logic in service handlers.

# Acceptance Criteria
- Repository interfaces and implementations exist for all core entity groups.
- Conversation and listing queries support the planned chatbot workflow.
- Tests cover create, update, lookup, and status transition persistence.

# Validation
- Run repository-level tests against a test database.
- Manually inspect that service code depends on repositories rather than raw queries.

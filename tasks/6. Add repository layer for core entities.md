# Why
Service code will need stable persistence APIs before the admin bot, chatbot service, and userbot worker can safely read or mutate database state. This task creates a repository layer so later tasks do not embed raw SQL directly inside business-flow modules.

# Context
- Use the existing Python stack in this repo: Python 3.10+, `psycopg` 3, `pytest`, and the dataclass-based domain models in [src/shared/domain/models.py](/home/ivan/work yourself/rental_helper/src/shared/domain/models.py).
- The database schema already exists in [src/shared/persistence/migrations/0001_initial_schema.up.sql](/home/ivan/work yourself/rental_helper/src/shared/persistence/migrations/0001_initial_schema.up.sql).
- Current behavior: callers would have to write SQL directly against tables such as `managed_accounts`, `bound_groups`, `conversations`, `listings`, `listing_offers`, and `viewing_appointments`.
- Expected behavior: `shared.persistence` exposes repository classes and mapping helpers that translate rows into domain models and centralize CRUD/query logic for core entities.
- Do not introduce an ORM. Keep the persistence layer thin and explicit around SQL, transactions, and domain model mapping.

  - `src/shared/persistence/repositories/__init__.py`
  - `src/shared/persistence/repositories/accounts.py`

  - `src/shared/persistence/repositories/_mappers.py`
  - `src/shared/persistence/repositories/_base.py`
- If a flatter module layout fits better with the existing package, keep everything under `src/shared/persistence/` but preserve clear boundaries by entity group.
- Reuse the existing domain types:
  - `ManagedAccount`
  - `BoundGroup`
  - `Conversation`
  - `LeadRequirements`
  - `Listing`
  - `ListingOffer`
  - `ViewingAppointment`
- Repositories should accept an existing `psycopg.Connection` or transaction-scoped cursor context rather than owning long-lived connections themselves.

# Constraints
- Repository APIs must be shaped around service use cases, not generic table dumps.
- Keep transaction boundaries explicit so future multi-step workflows can compose several repository calls in one database transaction.
- Keep SQL parameterized with `psycopg`; no string-built SQL for dynamic values.
- Domain model invariants from [src/shared/domain/models.py](/home/ivan/work yourself/rental_helper/src/shared/domain/models.py) must remain the single source of truth for validation.
- Preserve schema assumptions already encoded in the migration, especially:
  - one bound group per managed account
  - one `(conversation_id, listing_id)` pair in `listing_offers`
  - appointment foreign keys tied to both conversation and listing ownership
- Do not duplicate repository query logic in service packages.

# Repository Behaviors To Cover
- Accounts and bound groups:
  - create and fetch `ManagedAccount`
  - update account status
  - attach or replace the single `BoundGroup` for an account
  - look up an account by Telegram phone and by account id
- Conversations:
  - create a conversation with `LeadRequirements`
  - fetch by conversation id
  - fetch active/open conversation by `(managed_account_id, telegram_user_id)`
  - update stage, status, requirements, and `handoff_to_human`
  - persist `updated_at` changes on every mutating operation
- Listings and listing offers:
  - create or upsert a listing indexed from a bound group message
  - fetch listings by account and search fields needed by the chatbot flow: district, room count, budget ceiling, active status
  - fetch a listing by `(bound_group_id, group_message_id)`
  - create a `ListingOffer`
  - mark an offer as rejected
  - list previously offered listing ids for a conversation in reverse `offered_at` order
- Viewing appointments:
  - create an appointment
  - fetch appointments by conversation id
  - fetch upcoming appointments for an account ordered by `starts_at`
  - update appointment status

# Acceptance Criteria

- Row-to-domain mapping is centralized instead of repeated across repository methods.
- Repository methods cover the CRUD and lookup paths required by planned tasks `10`, `16`, `17`, `19`, and `20`.

- Tests verify listing search queries exclude archived listings and respect budget and room-count filters.
- Tests verify duplicate offer protection and appointment ordering behavior.

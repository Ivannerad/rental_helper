# Why
The userbot worker and chatbot service need stable, typed message schemas before queue consumers are built.

# Context
- Lives in shared queue schema modules.
- Current behavior: inbound and outbound event shapes are only described in prose.
- Expected behavior: typed payloads define message structure, required fields, versioning, and idempotency keys.

# Constraints
- Contracts must support inbound messages, outbound actions, and dead-letter/error events.
- Event shapes must be serializable without service-specific objects.
- Preserve correlation IDs across all events.

# Acceptance Criteria
- Schema definitions exist for inbound Telegram messages, outbound chatbot actions, and error events.
- Event models include account ID, user ID, correlation ID, and idempotency data where required.
- Serialization and deserialization tests pass.

# Validation
- Run unit tests for event schema validation.
- Manually inspect sample serialized payloads for completeness and clarity.

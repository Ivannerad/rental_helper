# Why
Durable storage is required for account state, conversations, listings, and viewing schedule data.

# Context
- Lives in persistence modules and migration files.
- Current behavior: no database schema exists.
- Expected behavior: PostgreSQL tables persist the full v1 state described in the plan.

# Constraints
- Keep migrations repeatable and safe for local re-creation.
- Preserve traceability between Telegram entities, conversations, and queue events.
- Avoid denormalization that makes scheduling or listing lookup ambiguous.

# Acceptance Criteria
- Initial migrations create all core tables and indexes.
- Foreign keys capture account, group, conversation, listing, and appointment relationships.
- Migration instructions are documented for local setup.

# Validation
- Run migration up/down or reset flow against a local PostgreSQL instance.
- Execute tests that verify schema creation and basic persistence operations.

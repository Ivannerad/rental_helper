# Why
Users must be able to review one listing at a time and continue searching when they reject an option.

# Context
- Lives in `src/chatbot_service/` search and offer policy modules.
- Current behavior: no listing selection logic exists.
- Expected behavior: the chatbot filters listings by requirements, presents ranked options, records rejections, and chooses the next best candidate.

# Constraints
- Use only listings from the bound group inventory in v1.
- Do not resend already rejected listings in the same conversation.
- When inventory is insufficient, escalate to manual review instead of inventing results.

# Acceptance Criteria
- Search logic filters by district, room count, and budget where available.
- Offer history prevents duplicate suggestions within a conversation.
- No-match behavior records unmet demand and produces the correct fallback action.

# Validation
- Run unit tests for ranking, rejection handling, and no-result fallback.
- Manually simulate a conversation that rejects multiple listings.

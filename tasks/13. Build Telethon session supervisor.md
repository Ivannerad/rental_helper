# Why
Managed accounts require a worker that can start, monitor, and reconnect multiple Telethon sessions safely.

# Context
- Lives in `src/userbot_worker/`.
- Current behavior: no userbot worker exists.
- Expected behavior: the worker loads enabled accounts, starts one Telethon client per account, and supervises reconnect behavior.

# Constraints
- Disabled accounts must not have active sessions.
- Session failures should be isolated so one account does not stop all others.
- Worker state should be observable through logs and status updates.

# Acceptance Criteria
- The worker loads enabled managed accounts from persistence.
- One supervised Telethon session is created per enabled account.
- Reconnect and shutdown handling are implemented for session failures.

# Validation
- Run tests for account loading and supervision logic.
- Manually start the worker with multiple test accounts and confirm independent session handling.
